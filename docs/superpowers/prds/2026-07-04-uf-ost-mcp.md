# PRD: uf-ost — SQLite OST Storage + MCP Server

Source of truth: `docs/superpowers/specs/2026-07-04-uf-ost-mcp-design.md` (10 locked
decisions) and `docs/superpowers/plans/2026-07-04-uf-ost-mcp-phases.md` (phase order +
origin-session facts). This PRD restates them from the user's perspective; it does not
reopen any decision. Triage label: `ready-for-agent`.

## Problem Statement

I use AI coding agents against real Salesforce orgs, but the agents can't see the org's
actual schema or Apex symbols. So they guess field API names, invent picklist values,
assume an object has a field it doesn't, or — worst — read a sandbox's shape while
writing production code. Salesforce's official DX MCP exists, but it queries live: the
ApexClass SymbolTable pull alone takes ~145 s on a large managed org, it floods the
agent's context with unfiltered output, and it ships no retrieval discipline, so the
agent doesn't know when its cached view is stale.

Separately, ultraforce already builds a good offline org index (OST: sObject schema +
Apex symbols), but it persists as 4–5k JSON files per org (plus a 57 MB raw-cache blob).
That layout can't be safely read while a reindex writes, holds ~100–200 MB of parsed
structs resident, has no full-text search, and leaked a real bug: the on-disk describe
JSON is raw SF **camelCase** while the index is snake_case, and the Node prototype
shipped a shape-assumption bug because of it.

## Solution

Ship `uf-ost`: move ultraforce's org index to **SQLite** (one `index.db` per org, WAL
mode), then expose it to agents through a native **MCP server** (`ultraforce`, tools
`ost_*`). The agent asks "what fields does Account have in org X?", "which orgs carry
`Custom_Field__c`?", "what's the picklist for Status?", "what does this Apex class
expose?" and gets token-lean, line-filtered answers in milliseconds — every response
stamped with the org and the snapshot's age, so the agent can tell fresh from stale.
Refresh is two-mode: a cheap synchronous delta (`ost_sync`) and an async singleton full
reindex (`ost_reindex`) that the agent never waits on. A shipped skill teaches the
retrieval discipline (verify freshness → sync → reindex + fall back to live `sf`). My own
omni-stack workspace migrates onto it, with the retired Node prototype serving as a
differential-test oracle.

Persistence lands first (Phase 1), the MCP bin second (Phase 2), the omni-stack migration
third and outside this repo (Phase 3).

## User Stories

1. As an AI coding agent, I want the exact field list (name, type, referenceTo, picklist
   flag, custom flag) for an object in a named org, so that I stop guessing API names.
2. As an AI coding agent, I want to ask which objects/orgs carry a given field, so that I
   can detect cross-org schema drift before writing code.
3. As an AI coding agent, I want the active picklist values for a field, so that I never
   invent an invalid value.
4. As an AI coding agent, I want an Apex class/interface/enum's member signatures from the
   offline index, so that I don't trigger the ~145 s live SymbolTable query.
5. As an AI coding agent, I want fuzzy full-text search over object/field names, labels,
   and Apex type names, so that I can find the right API name when I only know a
   human-ish term.
6. As an AI coding agent, I want per-org freshness, counts, any stdlib error, and whether
   a reindex is in progress, so that I can judge how much to trust the index.
7. As an AI coding agent, I want every response stamped with the org and snapshot age, so
   that I can't silently mix a sandbox's schema into production code.
8. As an AI coding agent, I want a cheap synchronous delta refresh returning
   `{added, updated, removed}`, so that I can reconcile a single stale answer without a
   full rebuild.
9. As an AI coding agent, I want a full reindex I can kick off without waiting, so that I
   can request freshness and keep working against live `sf` in the interim.
10. As an AI coding agent, I want a skill that tells me when to consult OST and how to
    handle staleness/contradiction, so that my retrieval behavior is disciplined by
    default.
11. As the ultraforce desktop app, I want to load the index by SELECTing into the same
    in-memory `Ost`/`SObjectSchema` structs, so that completions and existing callers are
    unchanged by the storage swap.
12. As the ultraforce desktop app, I want the org indexed into SQLite and completions to
    keep working, so that the migration is invisible to end users.
13. As a reader of the index, I want WAL concurrency so I see a consistent snapshot while
    a background reindex writes, so that I never read a half-written, mixed-generation
    index.
14. As an operator, I want a full reindex to commit in a single transaction, so that there
    is no lockfile/tmp-rename machinery and no partial-generation window.
15. As an operator, I want one `index.db` per org under a shared root, so that dropping an
    org is a single file delete and adding one is trivial.
16. As an operator, I want to run `uf-ost index --org <alias>` headlessly from
    launchd/cron, so that snapshots stay fresh without the desktop app open.
17. As an operator, I want `--sync` on the indexer for watermark deltas, so that scheduled
    refreshes are cheap.
18. As an operator, I want `uf-ost status` to dump freshness, counts, stdlib_error, and
    running-reindex state, so that I can diagnose staleness from the shell.
19. As an operator, I want the snapshot root resolved as `--root` > `UF_OST_ROOT` env >
    `default_index_root()`, so that I can relocate storage without code changes.
20. As an operator, I want `ost_reindex` to be a global singleton, so that a second
    reindex request returns `already_running` instead of duplicating a heavy API pull.
21. As an ultraforce maintainer, I want the `fetch_apex_symbols` call to carry its own
    extended timeout in `acquire.rs` (like completions 300 s / describe 180 s), so that
    the ~145 s ApexClass query stops being killed by the 120 s invoker default — and the
    bin-level 300 s workaround (and the old `ost-index` bin) can be deleted.
22. As an ultraforce maintainer, I want a stdlib Tooling error (e.g. a managed package's
    bad `@AuraEnabled`) recorded in `meta.stdlib_error` and surfaced, so that an empty
    stdlib namespace is never silent.
23. As an ultraforce maintainer, I want the raw fetch responses (the 57 MB
    `org_types.json`) stored in a `raw_cache` blobs table, so that they're out of every
    query surface.
24. As an ultraforce maintainer, I want reads/writes to go through the typed structs
    end-to-end, so that the camelCase-vs-snake_case shape bug class cannot recur.
25. As an ultraforce maintainer, I want FTS5 tables (`fields_fts`, `apex_fts`) built with
    the index, so that `ost_search` comes for free.
26. As an ultraforce maintainer, I want no JSON→SQLite importer (absent `index.db` ⇒
    reindex, old JSON ignored), so that migration carries no throwaway reader code.
27. As an ultraforce maintainer, I want Phase 2 to never read JSON, so that there's no
    dead compatibility path once storage flips.
28. As the omni-stack owner, I want `.mcp.json` pointed at `uf-ost serve` and the private
    skill swapped to the MCP version, so that my daily agent workflow runs on this server.
29. As the omni-stack owner, I want a differential check where `ost.mjs` (JSON) and the
    MCP (SQLite) agree on the same org's outputs, so that I trust the migration before
    retiring the prototype.
30. As the omni-stack owner, I want launchd `com.dormon.ost-sync` switched to
    `uf-ost index --sync` per org, so that scheduled deltas run on the new indexer.
31. As the omni-stack owner, I want `db/ost` and the `scripts/ost/` prototype retired
    (keeping `ost.mjs` only until the diff passes), so that there's one source of truth.
32. As a prospective user reading the README, I want the storage location, org-IP
    sensitivity, and reindex API-consumption notes, so that I understand the cost and
    sensitivity before I run it.
33. As the repo owner, I want snapshot data (any org describe output) never committed, so
    that org IP never lands in git.

## Implementation Decisions

**Storage (Phase 1) — `apex-lang/src/snapshot.rs`, `sf-schema/src/store.rs`,
`apex-lang/src/acquire.rs`:**
- SQLite is canonical storage for both the app and the MCP (user override). New dep:
  `rusqlite` (bundled) only.
- One DB per org: `<root>/<alias>/index.db`, WAL mode. Org key = **sf alias**. Root =
  `default_index_root()`, overridable by `--root`/`UF_OST_ROOT`.
- Full reindex = one transaction. No JSON importer; absent `index.db` ⇒ reindex; old JSON
  snapshots ignored.
- In-memory `Ost` / `ApexType` / `SObjectSchema` structs and every consumer (desktop app,
  completions) are unchanged. `save_snapshot`/`load_snapshot` and `SchemaStore` gain
  SQLite backends; load = SELECT into the same structs.
- Delta path: preserve the `sync_org` guard — delete-reconcile only when **both** name
  lists are non-empty; watermark advances; upserts applied.
- Schema v1 (details refined in the plan): `meta` (schema_version, alias, org_id,
  api_version, indexed_at watermark, generation, nullable `stdlib_error`); `apex_types`
  (name, kind, namespace NULL=org else stdlib, parent_class, interfaces JSON, enum_values
  JSON); `apex_members` (type rowid, kind, name, return/prop type, params JSON,
  is_static); `objects` (name, label, label_plural, key_prefix, custom); `fields` (object
  rowid, name, label, type, custom, nillable, reference_to JSON, relationship_name,
  picklist JSON `[{label,value,active,defaultValue}]`); `raw_cache` blobs; FTS5
  `fields_fts(object_name, field_name, field_label)` + `apex_fts(type_name)`.
- Two upstream fixes fold into Phase 1: (a) `fetch_apex_symbols` gets
  `run_raw_with_timeout` with an extended timeout; the bin-level 300 s workaround and the
  `ost-index` bin's timeout code are removed. (b) stdlib fetch fails loud into
  `meta.stdlib_error` instead of yielding silent-empty namespaces.

**MCP bin (Phase 2) — new crate `crates/uf-ost/`:**
- Single binary, subcommands `serve` / `index --org <alias> [--sync] [--root <dir>]
  [--policy all]` / `status [--org <alias>]`. `index` absorbs
  `crates/features/src/bin/ost-index.rs`; the old bin is deleted.
- `serve` = MCP over stdio via `rmcp` (pin exact pre-1.0 version). Opens per-org DBs
  lazily, read-only.
- MCP server name `ultraforce`; 8 tools `ost_*`. Query: `ost_object`, `ost_field`
  (omit org ⇒ all orgs, drift check), `ost_picklist`, `ost_apex` (stdlib best-effort per
  `stdlib_error`), `ost_search` (FTS5), `ost_status`. Refresh: `ost_sync` (synchronous
  watermark delta ⇒ `{added, updated, removed}`), `ost_reindex` (async global singleton
  via lock row/file ⇒ `started` | `already_running`; progress on `ost_status`; no job-id
  system).
- Every tool response stamped with `org` + snapshot age (from `meta.indexed_at`).
- Skill shipped in-repo at `skills/ost/SKILL.md` (skills-CLI installable), content per
  spec §6. README section added.

**omni-stack migration (Phase 3, outside this repo):**
- `.mcp.json` → `uf-ost serve`; private skill rewritten to the in-repo MCP skill;
  differential check `ost.mjs` (JSON) vs MCP (SQLite) on the same org; launchd
  `com.dormon.ost-sync` → `uf-ost index --sync` per org (org list stays in the wrapper);
  `db/ost` + `scripts/ost/` retired, `ost.mjs` kept only until the diff passes.

**Global constraints:** never `git push` (local commits only); conventional commits with
`--no-gpg-sign` authorized per-invocation (never touch `.git` config); never commit
snapshot/describe data; MIT license repo-wide.

## Testing Decisions

Good tests here assert **external behavior** through the highest existing seam, not
storage internals. The preferred seams, one per layer:

- **Phase 1 seam — snapshot save/load round-trip.** The in-memory structs are unchanged,
  so the single highest seam is: build an `Ost`/`SObjectSchema` in memory → `save` to
  SQLite → `load` → assert struct equality. This is the existing snapshot round-trip seam;
  add SQLite as the backend under the same test shape rather than inventing a new seam.
  Covers schema fidelity (picklist/reference_to JSON columns), the sync_org delta guard
  (both-non-empty delete-reconcile; watermark advance), and `stdlib_error` population on
  an erroring org (assert non-empty instead of silent-empty namespaces).
- **Phase 2 seam — MCP stdio contract.** Drive the 8 tools through a scripted `rmcp`
  client / test harness over stdio and assert response shapes, including the mandatory
  org + snapshot-age stamp on every response, and the reindex singleton (second
  `ost_reindex` ⇒ `already_running`). Test at the tool boundary, not the DB.
- **Phase 3 seam — differential oracle.** `ost.mjs` reading the verified 2026-07-03
  SFDC_Staging JSON snapshot in `omni-stack/db/ost/` is the ground truth; assert MCP
  (SQLite) query outputs agree on the same org without re-indexing.

Prior art: the existing crate tests in `apex-lang`/`sf-schema` (must stay green — the gate
for Phase 1) and the desktop indexing + completions flow, which must work end-to-end
against SQLite. Keep tests to one runnable check per non-trivial path; no new frameworks.

## Out of Scope

GUI toggle/panel; job-id system (superseded by the singleton design); benchmark tables;
multi-client presets; marketplace/registry listings; scheduled drift alerting (passive
cross-org queries via `ost_field` already cover the need); support commitments and
multi-client docs in v0; any JSON→SQLite importer or throwaway JSON reader; live/no-cache
query mode. Phase 3 executes outside this repo and is coordinated with the user, not
delivered by this repo's phases.

## Further Notes

- **Honest basis:** JSON perf is already fine (42 MB `index.json` parses in 86 ms, full
  field scan 8 ms). SQLite's win is WAL concurrency + low resident memory + free FTS5 +
  no 4–5k small files — **not** speed. Do not build perf machinery.
- **Known/open items:** the desktop app may key snapshots by username rather than alias —
  v0 convention is "operate by alias", unification is a follow-up. `rmcp` is pre-1.0: pin
  exact version, expect breaking bumps. Multi-org confusion (agent reads sandbox, writes
  prod) is a real hazard, mitigated by mandatory org stamping. Reindex consumes API
  (~4–5k describes + the heavy ApexClass query) — note in README.
- **Sequencing:** each phase lands independently; Phase 2 never reads JSON.
- **Positioning:** self-use first, published as-is. Differentiators vs Salesforce DX MCP:
  token-lean line-filtered output, offline Apex SymbolTable index, shipped retrieval-
  discipline skill.
